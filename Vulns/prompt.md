漏洞原理笔记生成提示词 v2

请根据我提供的漏洞主题或原始资料，帮我整理一篇中文版网络安全漏洞笔记。

这篇笔记不是课程翻译，不要复述原文，不要保留口语化表达。你需要把资料中的知识点提炼成网络安全工程师能理解、能复习、能用于 GitHub 项目沉淀的结构化笔记。

---

## 1. 内容定位

这篇笔记主要用于：

- 我自己复习漏洞原理；
- 后续放进 GitHub 项目；
- 我会自行穿插 PortSwigger Lab 或其他靶场复现；
- 你只负责漏洞原理、成立条件、关键机制、风险边界、防护原则和必要的面试概括。

不要写测试手册。  
不要写靶场规划。  

但笔记不能过于干瘪。

在不破坏主线结构的前提下，可以加入少量典型样例、关键 payload、请求片段、文件名样例、判断链路或对比表，用来帮助理解漏洞机制。

这些样例的作用是“辅助理解机制”，不是堆技巧清单。

如果需要提到 PortSwigger Lab，只说明该 Lab 对应哪类机制或章节位置。

例如：

“该机制可对应布尔盲注类 Lab，适合插入到盲注判断一节。”

不要写具体操作步骤，不要把 Lab 名称变成正文主线。

### 资料来源优先级

如果需要补充资料或校准术语，优先参考：

1. 官方文档、官方安全公告、厂商修复说明；
2. OWASP WSTG、OWASP Cheat Sheet、MITRE CWE；
3. PortSwigger Web Security Academy；
4. NVD / CVE / CNVD 等漏洞库；
5. 可信安全厂商或研究员技术分析；
6. 我提供的原始资料。

不要只依赖单一博客。  
如果不同来源说法冲突，以官方公告、标准资料和可验证实验结果为准。  
如果某个结论不确定，应明确说明，不要编造。

## 2. 结构要求

不需要所有漏洞都强行使用完全相同的模板，但必须保证结构清晰、没有多余标题、没有碎片化分条。

每篇笔记只保留一个主标题，下方根据漏洞特点设计 5-8 个主体章节。

标题必须服务于理解漏洞本质，不能为了完整而机械堆章节。

例如文件上传漏洞可以使用：

# 04S - 文件上传漏洞

## 1. 漏洞本质

## 2. 风险成立条件

## 3. 上传链路与服务端解析机制

## 4. 校验机制缺陷

## 5. 典型绕过与风险边界

## 6. 防护原则

## 7. 面试概括

但其他漏洞不必照抄这 7 个标题。

例如：

SQL 注入可以重点写：

- SQL 查询拼接机制；
- 注入类型；
- 数据提取方式；
- 盲注判断；
- 防护原则。

SSRF 可以重点写：

- 服务端请求边界；
- 内网访问；
- URL 解析差异；
- 云元数据风险；
- 绕过方式；
- 防护原则。

访问控制可以重点写：

- 认证与授权的区别；
- 对象级权限；
- 水平越权；
- 垂直越权；
- 功能级越权；
- 防护原则。

核心原则：

> 标题必须跟漏洞自身逻辑匹配，而不是套模板。

如果结构不适合当前漏洞，可以主动调整章节，但只需用 1-2 句话说明为什么这样排，不要输出冗长说明。

## 3. 写作风格

请使用中文，表达要像安全工程师写给自己看的原理笔记。

要求：

- 准确；
- 克制；
- 不口语化；
- 不写废话；
- 不堆大段背景；
- 不机械编号；
- 不把所有知识都塞进 1、2、3、4、5 的条目；
- 尽量用自然段解释核心逻辑；
- 只在需要对比、归纳时使用表格或简短列表。

不要写成：

“首先我们要知道……”  
“这个漏洞非常有意思……”  
“接下来让我们看看……”  
“攻击者可以轻松地……”  
“只需要一个 payload 就能……”

要写成：

“该漏洞的本质是……”  
“风险成立依赖……”  
“防护重点在于……”  
“该绕过方式利用的是……”  
“该样例只用于说明机制，不代表所有环境通用。”

同时要避免写得像安全规范条文一样干。

如果某个机制比较抽象，应加入一个短小、准确、贴合当前知识点的小例子帮助理解。

例子要服务于机制解释，不要为了显得内容多而硬塞。

---

## 4. 内容深度

每篇笔记控制在适合复习的长度，可以少，但不能缺少关键知识面。

必须覆盖：

- 漏洞本质；
- 风险成立条件；
- 关键机制；
- 常见成因；
- 典型风险；
- 核心绕过或变体；
- 防护原则；
- 面试概括。

如果某个漏洞存在特殊机制，应优先讲机制。

例如：

- 文件上传要讲上传链路、文件名处理、服务端解析、上传目录执行权限；
- SQL 注入要讲 SQL 语义被改变、查询上下文、数据提取方式、盲注判断；
- SSRF 要讲服务端请求边界、内网资源、URL 解析差异、云元数据风险；
- XXE 要讲 XML 外部实体、DTD、解析器配置；
- 反序列化要讲对象生命周期、魔术方法、gadget chain；
- 访问控制要讲认证和授权的区别、对象级权限、水平/垂直越权；
- SSTI 要讲模板引擎解析上下文和表达式执行边界；
- 路径遍历要讲路径拼接、路径规范化、目录边界、文件系统解析；
- 请求走私要讲前后端服务器对请求边界解析不一致；
- 原型污染要讲对象属性查找链、污染点、sink 点和影响边界。

## 5. 样例与 payload 使用规则

允许加入典型 payload 或样例，但必须遵守以下规则。

### 可以加入

用于解释机制的典型样例，例如：

- 文件上传中的双后缀、大小写、尾点、`.htaccess`、`::$DATA`、`%00`；
- SQL 注入中的 `' OR '1'='1`、`UNION SELECT`、布尔差异、时间延迟样例；
- SSRF 中的 `127.0.0.1`、内网地址、云元数据地址；
- XXE 中的 `DOCTYPE`、`ENTITY`、外部实体引用；
- XSS 中的 HTML 上下文、属性上下文、JavaScript 上下文样例；
- 路径遍历中的 `../`、绝对路径、双重编码；
- 反序列化中的对象生命周期、魔术方法触发点示例；
- SSTI 中的模板表达式识别样例；
- 访问控制中的用户 ID、对象 ID、角色字段差异样例。

这些样例应该放在对应原理段落下，作为“典型样例”或“理解样例”。

### 样例必须带边界说明

每个典型样例最好说明它的适用条件，而不是写成通用结论。

例如：

- `%00` 截断应说明依赖旧组件或特定底层处理，不是现代通用绕过；
- `::$DATA` 应说明属于 Windows NTFS ADS 场景，不适用于 Linux；
- `.htaccess` 应说明依赖 Apache 目录级配置生效；
- 图片马应说明通常需要配合文件包含、解析差异或后处理流程，单纯访问图片通常不会执行其中脚本；
- 时间盲注样例应说明依赖数据库函数、响应时间稳定性和服务端执行延迟；
- SSRF 云元数据样例应说明依赖云环境和网络可达性；
- XSS payload 应说明上下文，不同上下文下可用形式不同；
- 访问控制样例应说明漏洞点在授权判断，而不是单纯修改 ID 本身。

目标是让笔记更好读，而不是制造错误泛化。

样例不能打断漏洞主线。若某个样例无法直接解释当前机制，应删除，不要为了丰富内容而保留。

每个主体章节最多保留 1-2 个典型样例。  
如果一个样例已经能说明机制，不要继续追加同类 payload。  
样例优先选择最能解释机制的最小化形式。

典型样例应尽量说明：

- 样例形式；
- 解释的机制；
- 适用边界。

不要只给 payload。解释为什么成立。

## 6. 不需要做的内容

除非我明确要求，否则不要加入：

- 大量 HTTP 请求包；
- 工具安装教程；
- GitHub 项目规划；
- 过长参考资料；

如果需要提到 Lab，只保留概念层面的对应关系，不写完整步骤。

可以写：

“该机制可对应 PortSwigger 中的布尔盲注类 Lab。”

不要写：

“第一步点击这里，第二步抓包，第三步修改为……”

---

## 7. 面试概括

每篇最后可以保留一个“面试概括”部分。

这个部分不是给 GitHub 正式展示用，而是给我自己面试前快速复习用。

要求压缩成：

- 一句话解释漏洞；
- 风险成立条件；
- 常见利用方向；
- 防护重点。

不要写太长，不要写成背诵稿。

示例格式：

## 7. 面试概括

**一句话解释：** ...

**风险成立条件：** ...

**常见利用方向：** ...

**防护重点：** ...

## 8. 质量标准

完成后请在内部自检，不需要在正文末尾输出冗长自检结果。

自检标准：

这篇笔记是否能让我快速理解漏洞本质？  
是否避免了原文翻译腔？  
标题是否过多？  
有没有多余章节？  
是否把测试方法和原理混在一起？  
是否有关键知识点缺失？  
是否适合后续插入 Lab 复现？  
是否适合放进 GitHub 项目？  
是否过于干瘪，缺少必要样例？  
是否又反过来堆了太多 payload，破坏了主线？  
样例是否都有适用边界？  
是否避免把特定环境下的技巧写成通用结论？

如果结构经过主动调整，只需在正文开头或结尾用 1-2 句话说明为什么这样排。
