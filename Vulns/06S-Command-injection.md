# 06 - 操作系统命令注入

## 1. 漏洞本质

操作系统命令注入，也称 OS Command Injection 或 Shell Injection，指应用程序把用户可控输入拼接进系统命令，并交给操作系统 Shell 或命令解释器执行，导致攻击者可以改变原始命令语义，执行额外的系统命令。PortSwigger 对该漏洞的定义也是：攻击者能够在运行应用的服务器上执行操作系统命令，严重时可导致应用及其数据被完全攻陷。

该漏洞的核心不是“用户输入危险字符”本身，而是：

```
用户输入
→ 被拼接进系统命令字符串
→ 命令交给 Shell 解析
→ Shell 元字符改变命令边界
→ 执行非预期系统命令
```

典型危险代码逻辑类似：

```
system("ping -c 4 " . $_GET["host"]);
```

如果 `host` 没有被严格限制，攻击者就可能让原本只应该作为参数的内容变成新的命令。MITRE CWE-78 将其归类为“未正确中和 OS 命令中特殊元素”，并指出常见场景包括应用执行固定程序，但把外部输入作为参数传入。

---

## 2. 风险成立条件

| 条件     | 说明                                                                             |
| ------ | ------------------------------------------------------------------------------ |
| 命令调用条件 | 后端代码调用了 `system`、`exec`、`Runtime.exec`、`ProcessBuilder`、`subprocess` 等系统命令执行接口 |
| 输入可控条件 | 用户输入进入命令字符串、命令参数、环境变量或间接配置                                                     |
| 解析条件   | 命令经过 Shell 或命令解释器解析，或参数未被安全隔离                                                  |
| 校验缺陷   | 没有白名单校验，或仅做了脆弱的字符替换                                                            |
| 权限影响   | 注入命令以 Web 应用进程权限执行                                                             |

需要注意：命令注入的实际危害受进程权限限制。如果 Web 服务以低权限用户运行，攻击影响会受限；如果服务以高权限运行，风险会明显扩大。

---

## 3. 常见触发位置

命令注入常出现在“业务功能需要调用系统工具”的位置，例如：

| 功能场景      | 后端可能调用                         |
| --------- | ------------------------------ |
| 网络诊断      | `ping`、`traceroute`、`nslookup` |
| 文件处理      | `tar`、`zip`、`convert`、`ffmpeg` |
| 系统运维面板    | `service`、`systemctl`、`docker` |
| 日志与备份     | `cp`、`mv`、`cat`、`grep`         |
| 图片 / 视频处理 | ImageMagick、FFmpeg 等外部命令       |

这类漏洞在“运维后台、网络设备 Web 管理界面、监控平台、CMS 插件、文件处理服务”里更常见。OWASP 也指出，命令注入通常发生在应用把不安全的用户数据传递给系统 Shell 时，攻击命令会以 vulnerable application 的权限执行。

---

## 4. 命令拼接与 Shell 元字符

命令注入依赖命令解释器对特殊字符的解析。常见分隔符包括：

| 字符          | 常见含义          | 适用环境           |
| ----------- | ------------- | -------------- |
| `&`         | 前后命令分开执行      | Windows / Unix |
| `&&`        | 前一个命令成功后执行后一个 | Windows / Unix |
| `           | `             | 管道，把前一个输出传给后一个 |
| `           |               | `              |
| `;`         | 顺序执行多个命令      | Unix           |
| 换行符         | 命令分隔          | Unix           |
| `` `cmd` `` | 命令替换          | Unix           |
| `$(cmd)`    | 命令替换          | Unix           |

典型理解样例：

```
原始命令：
ping -c 4 127.0.0.1

用户输入：
127.0.0.1; whoami

实际语义：
ping -c 4 127.0.0.1
whoami
```

这个样例只用于说明命令边界如何被改变。实际测试时要根据系统类型、Shell 类型、输入所在位置和过滤情况判断可用形式，不能把某个分隔符当成所有环境通用。

---

## 5. 回显型、盲注型与带外型

命令注入可以按结果是否直接回显分成几类。

### 5.1 回显型命令注入

回显型最直观，命令执行结果会出现在 HTTP 响应中。例如注入低风险命令后，页面返回当前用户、系统信息或固定字符串。

常见验证思路：

```
输入点可控
→ 注入最小化命令
→ 响应中出现命令输出
→ 判断命令执行成功
```

这种场景适合插入 PortSwigger 基础命令注入 Lab，但笔记中不需要写具体 Lab 步骤。

### 5.2 盲命令注入

盲注型不会直接返回命令输出，需要通过时间、文件写入或外带通道判断。PortSwigger 的资料中也将盲 OS 命令注入分为时间延迟、输出重定向和 OAST 带外交互等验证方式。

典型判断链路：

```
注入延迟命令
→ 响应时间明显变长
→ 推断命令被执行
```

边界说明：时间型判断依赖网络稳定性和服务端负载，不能只看一次请求结果。应该对比正常请求和注入请求的响应时间差异。

### 5.3 输出重定向

如果命令输出不回显，但 Web 根目录可写，可以尝试把输出写入静态文件，再通过浏览器访问验证。

```
命令执行
→ 输出写入 Web 可访问目录
→ 浏览器访问文件
→ 读取执行结果
```

边界说明：这种方式依赖文件系统权限和 Web 目录可写性。没有写权限时不成立。

### 5.4 OAST 带外验证

如果目标能访问外部网络，可以通过 DNS 或 HTTP 回连判断命令是否执行。OWASP WSTG 中也将命令注入测试描述为通过 HTTP 请求向应用注入 OS 命令，并观察应用或外部交互结果。

边界说明：OAST 依赖目标服务器能访问外网和 DNS 出口，不适合默认假设所有内网环境可用。

---

## 6. 命令注入与参数注入的区别

命令注入通常是直接让 Shell 执行额外命令；参数注入则不一定执行新命令，而是改变原本程序的参数语义。

例如：

```
tar -cf backup.tar 用户输入
```

如果用户输入变成某些特殊参数，可能不会执行新的系统命令，但会改变 `tar` 的行为。OWASP Cheat Sheet 也强调，防御时不能只盯着 Shell 分隔符，外部输入作为参数时同样需要严格处理。

简单区分：

| 类型      | 重点                 |
| ------- | ------------------ |
| OS 命令注入 | 改变命令边界，执行额外 OS 命令  |
| 参数注入    | 不一定新增命令，但改变原程序参数行为 |
| 共同点     | 都源于用户输入进入命令执行边界    |

---

## 7. 防护原则

最有效的防护是 **避免从应用层直接调用系统命令**。OWASP OS Command Injection Defense Cheat Sheet 明确建议优先使用语言内置 API 或安全库函数替代 OS 命令，例如使用 `mkdir()` 这类平台 API，而不是拼接 `system("mkdir ...")`。

如果确实必须调用外部命令，应遵循以下原则：

| 防护点      | 说明                               |
| -------- | -------------------------------- |
| 避免 Shell | 使用参数数组形式调用进程，避免 `sh -c`、`cmd /c` |
| 固定命令     | 命令本身不能由用户控制                      |
| 参数白名单    | 对用户输入进行允许列表校验                    |
| 类型约束     | 数字只允许数字，枚举值只允许预定义选项              |
| 最小权限     | Web 进程使用低权限账户运行                  |
| 输出隔离     | 不直接向用户返回敏感命令结果                   |
| 日志记录     | 记录异常参数和命令执行错误                    |
| 不依赖黑名单   | 不把过滤 `;`、`&` 当成核心防护              |

错误做法：

```
把危险字符替换掉
只过滤分号
只过滤空格
只过滤某几个命令名
```

这种黑名单容易遗漏不同系统、不同 Shell、不同编码和不同调用方式下的变体。防护重点应该是 **不让用户输入进入命令语义边界**，而不是和攻击者比赛谁记得更多特殊字符。
