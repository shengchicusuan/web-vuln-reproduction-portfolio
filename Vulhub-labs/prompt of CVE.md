## 3. 浏览器、Burp 与工具使用规则

复现文档默认遵循：

> 能用浏览器访问的页面，尽量用浏览器验证。
> 只有真正需要修改请求路径、参数、Header、multipart 表单、编码内容或特殊字段时，才使用 Burp。
> 不要把全文写成“请求包复现教程”。

具体要求：

### 浏览器负责

- 访问靶场首页；
- 确认服务是否启动；
- 进行普通业务操作；
- 验证上传、登录、访问页面、结果展示；
- 验证漏洞成功后的页面效果。

### Burp 负责

- 捕获正常业务动作；
- 修改关键路径、参数、Header、Cookie、multipart 字段或编码内容；
- 触发浏览器无法直接构造的 HTTP 层漏洞；
- 展示最关键的一处修改点即可。

### 非 HTTP 协议特殊处理

如果漏洞不是普通 HTTP 协议触发，例如：

- AJP；
- Redis；
- Dubbo；
- RMI；
- gRPC；
- FastCGI；
- WebSocket 特殊帧；
- 二进制协议；
- 消息队列协议；

不要硬说可以用 Burp Repeater 直接复现。

应明确说明如下例子：

- 浏览器只能验证 Web 页面或 HTTP 服务状态；
- Burp 只能辅助记录普通 HTTP 流量；
- 真正漏洞触发需要使用对应协议工具或专用客户端；
- 不要把 Burp 当成万能复现工具。

例如 AJP 漏洞应写：

“浏览器和 Burp 只用于确认 Tomcat HTTP 服务正常；真正漏洞触发发生在 AJP 8009 端口，需要使用 AJP 专用工具。”

不一定要按照例子来处理，可自行决定。

## 4. 文档结构要求

文档标题格式：

# CVE-XXXX-XXXX - 漏洞名称复现

主体章节建议使用 8-11 个二级标题，不要过度拆碎。

优先使用以下结构，但可以根据漏洞特点调整：

## 1. 漏洞概述

说明漏洞是什么、属于什么类型、影响什么组件、风险是什么。

不要大段背景介绍。
不要写“这个漏洞非常有意思”之类口语化表达。

## 2. 影响版本与利用条件

列出影响版本、修复版本、漏洞成立条件。

建议用表格压缩：

| 条件     | 说明  |
| ------ | --- |
| 版本条件   | ... |
| 功能条件   | ... |
| 配置条件   | ... |
| 权限条件   | ... |
| RCE 条件 | ... |

如果 RCE 依赖额外条件，必须写清楚。  
不要把“可能 RCE”写成“必然 RCE”。

## 3. 漏洞原理

用自然段解释漏洞链路。

必须讲清楚：

- 用户可控点是什么；
- 框架 / 组件处理了什么；
- 安全边界在哪里失效；
- 为什么会造成文件读取、路径穿越、命令执行、越权或其他影响。

可以加入短的理解链路，例如：

用户输入
→ 框架解析
→ 安全校验失效
→ 进入危险函数 / 危险路径
→ 造成漏洞影响

如果需要样例，可以给少量关键样例，但不能堆 payload。

## 4. Vulhub 环境启动

写清楚进入目录、启动环境、访问地址。

示例格式：

cd vulhub/xxx/CVE-XXXX-XXXX
docker compose up -d

环境启动后，浏览器访问：

http://127.0.0.1:端口

说明看到什么页面代表环境正常。

不要写 Docker 教程。
不要解释 Docker 基础概念。

## 5. 浏览器确认基础功能

用浏览器完成可视化验证，例如：

- 首页是否正常；
- 登录是否成功；
- 上传页面是否存在；
- 默认页面是否可访问；
- 普通业务流程是否正常；
- 普通访问是否失败，用作漏洞对照。

## 6. 使用 Burp 或对应工具触发漏洞

如果是 HTTP 漏洞：

- 说明浏览器先完成正常操作；
- 然后在 Burp 中修改关键字段；
- 只展示关键路径、关键参数、关键 Header 或 multipart 片段；
- 说明修改点为什么能触发漏洞。

如果是非 HTTP 协议漏洞：

- 明确说明 Burp 不能直接触发；
- 浏览器 / Burp 只负责辅助确认；
- 真正触发使用对应协议工具；
- 写清楚工具参数对应含义，不要只丢命令。

## 7. 浏览器验证漏洞结果

尽量用浏览器验证最终效果，例如：

- 访问上传后的文件；
- 查看页面回显；
- 查看敏感文件读取结果；
- 验证越权后的页面状态；
- 验证命令执行返回的安全测试输出。

如果不能用浏览器验证，则说明原因，并使用工具输出作为证据。

## 8. 结果判断

用表格说明不同现象代表什么。

例如：

| 现象                   | 含义          |
| -------------------- | ----------- |
| 返回目标文件内容             | 文件读取成功      |
| 访问目标页面显示 hello world | 文件写入并被解析    |
| 返回 403               | 可能被访问控制拦截   |
| 返回 404               | 路径、环境或版本不符合 |
| 返回源码                 | 文件上传成功但未被解析 |
| 连接失败                 | 端口未开放或协议不通  |

结果判断要服务于复现，不要写成空泛总结。

## 9. 修复建议

优先写官方推荐修复方式。

必须覆盖：

- 升级到修复版本；
- 禁用或限制危险功能；
- 关键配置加固；
- 输入校验或路径规范化；
- 权限隔离；
- 业务层防护原则。

如果官方明确给出修复版本，应写清楚。

不要写空话，例如“加强安全意识”。

## 10. 复现总结

用 1-3 段总结：

- 漏洞触发入口；
- 漏洞本质；
- 复现成功的关键点；
- 风险升级条件；
- 该 CVE 适合沉淀到哪个漏洞专题中。

不要写成面试稿。
不要写成心得体会。

## 13. 特殊情况处理

### 如果漏洞是文件上传类

必须区分：

- 普通上传是否成功；
- 上传目录是否可执行；
- 文件是否逃逸预期目录；
- 文件是否被服务端解析；
- RCE 是否依赖额外条件。

不要把“上传成功”直接等同于“RCE 成功”。

### 如果漏洞是路径遍历类

必须讲清楚：

- 路径入口；
- 路径规范化问题；
- 目录边界；
- 访问控制条件；
- 读取型和写入型风险区别。

### 如果漏洞是访问控制类

必须讲清楚：

- 认证和授权的区别；
- 水平越权 / 垂直越权 / 功能级越权；
- 用户身份、对象 ID、角色权限之间的关系；
- 成功判断不只看状态码，还要看业务结果。

### 如果漏洞是反序列化类

必须讲清楚：

- 反序列化入口；
- 对象生命周期；
- 魔术方法或 gadget chain；
- sink 点；
- 为什么能从对象恢复走到危险行为。

### 如果漏洞是 SSRF 类

必须讲清楚：

- 服务端请求边界；
- 内网资源；
- URL 解析差异；
- 云元数据风险；
- 回显型、盲打型和二次利用差异。

### 如果漏洞是非 HTTP 协议类

必须明确说明：

- 浏览器能验证什么；
- Burp 能辅助什么；
- 真正触发工具是什么；
- 协议端口和 HTTP 页面不是同一个入口。
